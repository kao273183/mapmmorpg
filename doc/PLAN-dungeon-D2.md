# 地城 D2 執行計畫：群系地形與事件變體

> 狀態：D2-A～D2-F 已實作，D2 完成。建立日期：2026-07-22，最近更新：2026-07-22。對應 [DESIGN-dungeon.md](DESIGN-dungeon.md) 的 D2 範圍。

## 1. 交付目標

D2 要在不改寫 D1 路線流程的前提下，交付兩層可重複擴充的內容：

1. 五個群系各一種可預警、可迴避、沒有無解配置的專屬地形。
2. 寶箱、祭壇、試煉三個事件家族共 12 種資料化結果。
3. 路線卡在進房前顯示實際的地形、事件、成本與最低獎勵，不出現預覽與內容不一致。
4. 同一地城種子與選擇序列，可重現房型、平台、地形、事件和敵人配置。

D2 不包含詛咒、鑰匙、隱藏房、Boss 重製、完整數據分析與大規模數值平衡；這些留給 D3 或後續版本。

## 2. 現況盤點與必要前置

現有 D1 已經具備 seeded 路線、`roomSpec`、雙路線卡、章節結算及撤退流程，可直接延伸。D2 開工前需補齊下列契約：

| 現況 | D2 調整 |
|---|---|
| `roomSpec` 只有房型、威脅與獎勵 | 進房前加入 `biomeId`、`hazardId`、`eventId`、`enemyTags` 與完整 `rewardTags` |
| D1 尚未加入「群系險境」房型 | 新增 `hazard` 房型、路線權重與探索評價，作為專屬地形的主要入口 |
| 事件在 `genFloor()` 內才選出 | 在 `makeRoomSpec()`／路線生成階段決定，`genFloor()` 只負責落地 |
| 平台與敵人仍大量使用 `Math.random()` | 每房建立獨立 RNG，平台、地形、事件位置及敵人配置依 `roomSpec.seed` 產生 |
| 事件 UI 寫死為兩個選項 | 改為讀取事件定義的動態選項，支援補給箱的三選一 |
| 房間完成主要依賴 `mons.length === 0` | 加入試煉狀態；房間完成要同時滿足敵人與必要事件條件 |
| 沒有地形生命週期 | 新增獨立的生成、更新、繪製、碰撞與清理入口 |

## 3. 核心資料契約

### 3.1 房間規格

```js
roomSpec = {
  id,
  floor,
  chapter,
  roomIndex,
  biomeId,
  type,
  threat,
  score,
  enemyTags: [],
  rewardTags: [],
  hazardId: null,
  eventId: null,
  seed
};
```

規則：

- 路線卡、`genFloor()`、HUD 與測試都只讀這份規格。
- `eventId` 與 `hazardId` 在顯示路線卡之前確定，進房後不得重新抽取。
- `treasure` 房只選寶箱事件；`event` 房選祭壇或試煉；`hazard` 房固定包含當前群系地形，第二章起的地形試煉可同時帶有 `eventId` 與 `hazardId`。
- 一般第一章房間不組合菁英與地形；第二章起每房最多兩個威脅標籤。
- 房間 RNG 依用途拆流，例如 `seed:platforms`、`seed:hazards`、`seed:enemies`，避免新增特效亂數改變關卡內容。

### 3.2 地形定義

`dungeon-data.js` 新增 `DUNGEON_HAZARD_DEFS`：

```js
hazardDef = {
  id,
  biomeId,
  name,
  previewTag,
  tutorial,
  warningFrames,
  activeFrames,
  cooldownFrames,
  maxPerRoom,
  minChapter,
  rewardBonus
};
```

執行期以 `dungeonHazards` 儲存實體，統一由下列入口管理：

- `spawnDungeonHazards(roomSpec, worldWidth, eventX, platforms)`
- `updateDungeonHazards()`
- `drawDungeonHazards()`
- `clearDungeonHazards()`

高傷地形共用 `warn → active → cooldown` 狀態，不各自在 `game.js` 內建立獨立計時方式。

### 3.3 事件定義與狀態

`dungeon-data.js` 新增 `DUNGEON_EVENT_DEFS`：

```js
eventDef = {
  id,
  family,
  name,
  description,
  minChapter,
  threat,
  costText,
  rewardTags,
  choices: [],
  effectId
};
```

事件定義維持純資料，實際效果由 `effectId` 對應到集中式處理函式，避免資料檔依賴載入順序。

`dungeonRun.eventHistory` 只在玩家進入選定房間時記錄事件 ID，未選擇的路線不計入歷史；最近 2 個事件權重為 0，再前 2 個降為 25%。執行中的試煉另存：

```js
floorTrial = {
  eventId,
  status,
  timeLeft,
  startHp,
  damageTaken,
  targetCount,
  rewardState
};
```

所有試煉都必須能進入 `success`、`failed` 或 `declined`，任何結果都不能讓出口永久關閉。

## 4. 五種地形首輪規格

| 群系 | 地形 | 首輪行為 | 配置安全限制 |
|---|---|---|---|
| 翠綠草原 | 荊棘根鬚 | 地面土痕預警 45 幀，刺出 18 幀，命中造成小傷害與 1 秒緩速 | 不放在入口／出口 180 px 內；相鄰根鬚間保留 120 px 安全帶 |
| 幽暗洞窟 | 落石區 | 落點框與碎屑預警 60 幀，石塊下落後消失 | 不鎖定靜止玩家必經點；同時最多兩個落點且不得覆蓋全部地面寬度 |
| 熾熱熔岩 | 熔岩噴口 | 預熱 45 幀、噴發 30 幀、冷卻至少 90 幀 | 每個噴口兩側各保留可站立區；事件物件附近 140 px 禁止生成 |
| 冰霜凍原 | 冰面 | 接地時加速度下降並保留滑行；反方向可煞車，衝刺維持控制權 | 平台邊緣最後 32 px 不結冰；第一個冰面前顯示短教學 |
| 虛空深淵 | 虛空平台 | 閃爍兩次後消失 90 幀，再固定返回 | 每段都必須有穩定替代路線；入口到出口不能只依賴消失平台 |

首輪傷害以玩家最大 HP 的低比例加最低值計算，並套用短無敵；同一地形的一次有效期只命中一次。確切傷害在 D3 依承傷紀錄調整。

地形房的額外獎勵先採房間結算加成，不直接提高怪物血量：

- 群系險境完成：額外靈魂或一份基礎材料。
- 地形試煉完成：稀有裝備與附魔塵。
- 第一章地形只出現在明確標示的險境／地形試煉，不混入安全戰鬥。

## 8. 驗收清單

### 自動檢查

- 產生至少 1,000 組種子，同一種子結果可重現。
- 每張路線卡的房型、威脅、地形、事件與最低獎勵等於實際 `roomSpec`。
- 最近兩個事件不會再出現；再前兩個事件權重正確下降。
- 第一章只出現允許的四個事件，不出現菁英＋地形組合。
- 每個地形避開入口、出口與事件互動區，並保留安全路徑。
- 十二事件與四試煉的獎勵最多發放一次。
- 試煉成功、失敗與拒絕後都能完成房間。

### 實機檢查

- 劍士與法師各完成五種地形測試房。
- 鍵盤、滑鼠及觸控各完成一次三選一事件。
- 故意讓限時、無傷、地形試煉失敗，確認出口與普通獎勵正常。
- 地形預警開啟／特效降低設定下仍可辨識危險判定。
- 低於 30% HP 時，Boss 前房仍至少有一條合理低風險路線。
- 死亡與撤退後，裝備、材料、靈魂及最高樓層保存正確。

## 9. 主要風險與控制方式

| 風險 | 控制方式 |
|---|---|
| 地形和平台組合造成無解房 | 先生成穩定主路徑，再放地形；生成後跑安全區與可達性檢查，不合法則退回無地形房 |
| 新亂數改變既有 seed 結果 | 依用途拆分 seed stream，視覺亂數不參與內容生成 |
| 試煉失敗後房間卡住 | 試煉使用單一狀態機，所有終止狀態都回到統一房間完成判斷 |
| `game.js` 再次膨脹 | 地形完整邏輯放入 `dungeon-hazards.js`；`game.js` 只加入呼叫接點 |
| 手機資訊過多 | 路線卡只顯示一個地形標籤、一個敵人標籤與兩個獎勵標籤；詳細成本放事件確認面板 |
| 12 種事件一次調試成本過高 | 先完成八個單次互動事件，再以共用試煉狀態機完成四個試煉 |
